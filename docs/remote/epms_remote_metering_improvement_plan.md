# EPMS 원격검침 개선안

## 1. 목적

현재 프로젝트의 원격검침 기능은 아래와 같은 기본 기능 중심으로 구성되어 있다.

- 매장 마스터 관리
- 매장-계측기 연결 관리
- 검침 시작/마감 스냅샷 생성
- 사용량 정산 및 청구

이 구조는 정산 서비스 관점에서는 유효하지만, 운영형 EPMS에서 기대하는 아래 기능까지 포함하기에는 부족하다.

- 원격검침 운영 대시보드
- 검침 이상 진단 및 예외 검증
- 매장별 피크 관리
- 경보, 조치, 제어 연계

이 문서는 현재 구조를 유지하면서도 운영성과 분석성을 강화할 수 있도록 원격검침 영역의 개선 방향을 정리한 것이다.

핵심 목표는 아래와 같다.

- 정산용 검침과 운영용 검침을 분리한다.
- 매장별 피크 관리 기능을 추가한다.
- 검침 실패, 누락, 역전, 이상치 검증 기능을 추가한다.
- 운영 화면과 정산 화면의 역할을 분리한다.
- 향후 자동 제어와 인센티브/패널티 연계를 고려한 데이터 구조를 마련한다.

## 2. 현재 구조 요약

현재 주요 테이블과 역할은 아래와 같다.

- `tenant_store`
  - 매장 마스터
- `tenant_meter_map`
  - 매장과 계측기 연결
- `billing_cycle`
  - 정산 주기
- `billing_rate`
  - 요금 기준
- `tenant_billing_contract`
  - 계약 정보
- `billing_meter_snapshot`
  - 시작/마감 시점 검침값 저장
- `billing_statement`
  - 청구 결과 저장

현재 정산 기준 흐름은 아래와 같다.

1. 매장 등록
2. 매장-계측기 연결
3. 정산 주기 선택
4. 시작 검침 스냅샷 생성
5. 마감 검침 스냅샷 생성
6. 사용량 계산
7. 청구서 생성

## 3. 현재 구조의 한계

### 3.1 운영 관점 한계

- 실시간 원격검침 운영 현황 화면이 없다.
- 검침 누락, 통신 장애, 계측기 이상, 역전 전력 등을 모아 보여주는 운영 화면이 없다.
- 계측기 교체, 매장 이동, 임시 연결 같은 예외 상황에 대한 검증이 약하다.

### 3.2 피크 관리 관점 한계

- 현재 피크 계산은 사실상 `MAX(active_power_total)` 기반이어서 실제 수요전력 관리용으로는 부족하다.
- 실제 요금/수요 관리에 중요한 `15분 평균 수요전력` 개념이 없다.
- 매장별 피크 한도, 경고 단계, 제어 기준을 관리하는 구조가 부족하다.
- 건물 전체 피크와 매장별 피크의 관계를 함께 분석하기 어렵다.

### 3.3 정산 관점 한계

- `billing_scope`가 있어도 청구 로직에서의 활용 범위가 제한적이다.
- 공용부 배분, 부분 개점/폐점, 임시 매장 같은 케이스를 충분히 설명하지 못한다.
- 검침 스냅샷과 운영 상태 이력이 분리되어 있지 않아 운영자가 원인 추적을 하기에 불편하다.

## 4. 개선 방향

개선 방향은 아래 네 축으로 나눈다.

1. 운영형 원격검침 대시보드
2. 정산용 검침 보정 체계
3. 피크 관리
4. 검침 예외 검증

각 축은 같은 계측 데이터를 보더라도 목적이 다르다.

- 운영형 원격검침은 지금 상태를 빠르게 본다.
- 정산용 검침 보정은 월 마감 값을 정확히 맞춘다.
- 피크 관리는 15분 수요를 예측하고 대응한다.
- 검침 예외 검증은 데이터 품질 문제를 조기에 확인한다.

## 6. 계산 로직 개선안

### 6.1 15분 수요전력 계산 추가

현재 최대 수요는 `active_power_total`의 단순 최대값에 가까워 실제 피크 관리용으로는 부족하다.

개선안:

- 측정 주기가 1분 이하이면 15분 롤링 평균을 계산한다.
- 매장별 `rolling_15m_kw`를 별도 집계한다.
- 청구용 최대수요는 `max(rolling_15m_kw)` 기준으로 계산한다.

필요 객체 예시:

- `vw_tenant_demand_15m`
- `sp_refresh_tenant_peak_log`

### 6.2 피크 예측 로직 추가

예측은 복잡한 AI가 아니어도 운영적으로 충분한 가치가 있다.

기본 방식:

- 현재 15분 구간까지의 누적 사용량
- 남은 시간 동안의 평균 사용량
- 종료 시점 예상 수요전력 계산

예시:

- 현재 9분 경과 시 평균 부하가 40kW
- 남은 6분도 비슷하면 종료 시 예상 15분 수요도 약 40kW

추후에는 시간대, 요일, 업종 계수 등을 반영할 수 있다.

### 6.3 청구용 피크 계산 분리

현재 청구 생성 과정에서 운영용 피크와 청구용 피크가 충분히 분리되어 있지 않다.

개선안:

- `billing_statement.peak_demand_kw`는 15분 기준으로 계산한다.
- 순간전력 기반 피크는 운영 모니터링 지표로 사용한다.
- 정산 피크와 운영 피크를 구분해 저장한다.

권장 컬럼:

- `instant_peak_kw`
- `billing_peak_demand_kw`

## 9. 피크 제어 연계안

### 9.1 제어 단계

모든 부하를 자동 차단하기보다는 현장 반발을 줄이는 점진적 구조가 바람직하다.

1. 알람만 발생
2. 운영자 확인 후 수동 제어
3. 조건 충족 시 자동 제어

### 9.2 제어 대상 정의

제어 대상 후보:

- 간판 부하
- 일부 공조기
- 환기 장비 일부
- 비필수 조명

피크 관리 정책과 제어 정책은 분리하되 서로 참조 가능해야 한다.

### 9.3 제어 이력 예시

테이블 예시: `peak_control_history`

권장 컬럼:

- `control_id`
- `store_id`
- `control_target`
- `control_type`
- `requested_at`
- `executed_at`
- `result_code`
- `result_message`
- `requested_by`

## 10. 운영 프로세스 개선안

### 10.1 일상 운영 프로세스

1. 원격검침 대시보드 확인
2. 미수신 또는 이상치 매장 확인
3. 현장 또는 설비 상태 확인
4. 필요 시 수기 보정 또는 조치

### 10.2 월 마감 프로세스

1. 정산 대상 주기 선택
2. 시작/마감 스냅샷 확인
3. 검침 이상 목록 검토
4. 필요한 보정 반영
5. 청구 생성
6. 정산 결과 검증

### 10.3 예외 처리 프로세스

- 계측기 교체
- 임시 점포 운영
- 부분 개점/폐점
- 공용부 사용량 분리
- 장기 통신 장애

예외 처리 프로세스는 업무 규칙과 이력 저장이 함께 설계되어야 한다.

## 11. 구현 우선순위

### Phase 1

- 원격검침 운영 대시보드
- 검침 이상 진단 테이블
- 피크 정책 테이블
- 기본 알람 연계

### Phase 2

- 15분 수요전력 집계
- 피크 로그 적재
- 매장별 피크 관리 화면
- 정산용 피크 계산 분리

### Phase 3

- 수기 보정 이력
- 공용부 배분 규칙 강화
- 월 마감 검증 프로세스 정비

### Phase 4

- 자동 제어 연계
- 인센티브/패널티 연계
- 고도화된 피크 예측

## 12. 바로 적용할 수 있는 수정 사항

즉시 적용 우선순위는 아래와 같다.

1. 검침 이상 탐지 테이블 추가
2. 피크 정책 테이블 추가
3. 15분 집계 뷰/프로시저 추가
4. 원격검침 운영 대시보드 추가
5. 정산용 피크 계산 분리

위 다섯 가지는 대규모 구조 변경 없이도 운영 품질을 크게 올릴 수 있다.

## 13. 결론

현재 원격검침은 `정산 가능한 구조`는 이미 갖추고 있다. 다만 운영형 EPMS로 발전하려면 아래 세 축이 반드시 더 필요하다.

- 원격검침 운영 대시보드
- 15분 수요전력 기반 피크 관리
- 검침 이상 검증 및 수기 보정 체계

다음 단계의 목표는 단순 청구 기능을 넘어서 아래 상태를 만드는 것이다.

- 매장별 현재 검침 상태를 실시간에 가깝게 파악할 수 있다.
- 월 마감 전에 검침 이상을 미리 정리할 수 있다.
- 매장별 피크를 예측하고 초과 전에 대응할 수 있다.
- 정산 결과가 운영 이력과 자연스럽게 연결된다.

현재 `peak_management.jsp`에는 아래 기능이 반영되어 있다.

- 순간 피크와 15분 수요전력 피크 동시 표시
- 매장별 피크 정책 상태 조회
- 경고/제어 대상 매장 목록
- 반복 초과 매장 우선순위
- 층/업종/상태 필터
- 정책 관리 화면 이동
- 정책 수정 및 대시보드 연동
- 집계 테이블 사용 여부 및 마지막 집계 시각 표시
- 집계 지연 경고 표시

관련 파일:

- `epms/peak_management.jsp`
- `epms/peak_policy_manage.jsp`
- `WEB-INF/src/epms/peak/PeakComputationService.java`
- `WEB-INF/src/epms/peak/PeakPolicyRepository.java`
- `WEB-INF/src/epms/peak/PeakPolicyManageServlet.java`

### Peak 정책 테이블

현재 매장별 피크 정책 관리가 구현되어 있다.

- 피크 한도
- 경고 기준
- 제어 기준
- 우선순위
- 자동 제어 사용 여부
- 적용 기간

적용 SQL:

- `docs/sql/src/create_epms_peak_policy_schema.sql`

### 15분 집계 테이블

현재 Peak 조회는 아래 경로를 지원한다.

1. `dbo.peak_15min_summary` 테이블이 있으면 집계 테이블 우선 사용
2. 없으면 `dbo.measurements`에서 조회 시점 기준으로 직접 15분 계산

적용 SQL:

- `docs/sql/src/create_epms_peak_15min_summary.sql`
  - `dbo.peak_15min_summary`
  - `dbo.sp_refresh_peak_15min_summary`

### 배치 운영 스크립트

15분 집계 테이블 운영을 위한 SQL Agent Job 스크립트가 추가되어 있다.

- `docs/sql/src/create_peak_15min_summary_agent_job.sql`

권장 적용 순서:

1. `create_epms_peak_15min_summary.sql` 실행
2. `create_peak_15min_summary_agent_job.sql` 실행
3. `peak_management.jsp`에서 집계 테이블 사용 여부와 마지막 집계 시각 확인
